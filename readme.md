# 第一阶段｜引擎运行基础骨架

阶段目标：彻底搞懂游戏程序最底层运行逻辑，掌握主循环、时间系统、静态渲染、输入交互四大基础能力，建立「循环驱动画面、时间驱动逻辑、输入控制交互」的核心认知。

### 第 1 课：最小游戏主循环实现（一切起点）

#### 学习目标

1. 掌握游戏主循环五部分：清屏 → 事件处理 → 逻辑更新 → 渲染绘制 → 屏幕刷新；

2. 理解双缓冲原理、画面残影成因、帧率控制；

3. 建立「循环驱动动态画面」的核心认知。

#### 重点理解

- 主循环五步顺序不能颠倒，尤其清屏必须在绘制之前，否则产生残影。

- 双缓冲：Pygame 中 `display\.flip\(\)` 交换前后台缓冲区，避免屏幕闪烁。

- 帧率控制：`pygame\.time\.Clock\(\)\.tick\(60\)` 使循环每秒最多执行 60 次，稳定游戏速度。

#### 动手练习题

1. 创建窗口 \+ 主循环 \+ FPS 显示 \+ 正常退出。

2. 循环内绘制固定方块，观察持续渲染。

3. 注释清屏代码，运行观察残影现象。

#### 🔁 映射引擎原理

- 在 Godot 脚本中打印 `\_process\(delta\)` 调用，思考引擎的主循环被隐藏在哪里。对比你手写的 `while running`。

### 第 2 课：时间系统与 delta time \+ 固定时间步长

#### 学习目标

1. 理解游戏逻辑必须与帧率解耦的原因；

2. 实现基于 `delta` 的移动、动画、计时；

3. 掌握固定时间步长原理，区分逻辑更新与渲染。

#### 重点理解

- delta time（增量时间）

    - 定义：上一帧到当前帧经过的秒数。

    - 公式：`位移 = 速度 × delta`。

    - 效果：无论帧率 30 还是 300，每秒移动总距离恒定。

- 固定时间步长（Fixed Timestep）

    - 问题：帧率波动时物理模拟（如跳跃高度）不稳定。

    - 解决：逻辑更新按固定间隔（如 1/60 秒）调用，渲染按实际帧率。

    - 引擎对照：Godot `\_process\(delta\)` （可变） vs `\_physics\_process\(delta\)` （固定）。

    #### 动手练习题

    1. delta 移动：方块以 300 像素/秒右移，帧率限制 30 或 300，观察每秒总位移不变。

    2. 固定时间步长：实现上面伪代码，在 `update\_physics` 中让方块匀速运动。

    3. 技能冷却：使用 `time\.time\(\)` 或累加 delta 实现 2 秒冷却计时器。

    #### 🔁 映射引擎原理

    - 在 Godot 中分别用 `\_process\(delta\)` 和 `\_physics\_process\(delta\)` 移动角色，打印调用次数。思考：为什么跳跃等物理代码必须放在 `\_physics\_process` 中？

### 第 4 课：输入系统与交互判定底层

#### 学习目标

1. 掌握键盘/鼠标的两种检测模式：事件触发 vs 状态轮询；

2. 手写实现长按移动、UI 点击、hover、拖拽；

3. 理解坐标转换、点\-矩形/点\-圆碰撞判定。

#### 重点理解

- 事件触发：`KEYDOWN`、`MOUSEBUTTONDOWN`，适合单次操作（菜单选择、跳跃）。

- 状态轮询：`key\.get\_pressed\(\)`、`mouse\.get\_pressed\(\)`，适合持续操作（移动、拖拽）。

- UI 交互本质：鼠标坐标 \+ 几何碰撞检测，无黑魔法。

#### 动手练习题

3. 可拖拽矩形：鼠标按下时选中，移动时更新位置。

#### 🔁 映射引擎原理

- 在 Godot 中用 `\_input\(event\)` 处理点击，用 `Input\.is\_action\_pressed\(\)` 处理长按。对比两种方式的适用场景。

### 第 13 课：粒子系统与批量特效底层原理

#### 学习目标

1. 理解粒子系统本质：批量生成短生命周期 \+ 运动 \+ 渐变；

2. 手写粒子类、发射器，实现火焰、爆炸、雪花特效；

3. 了解 Pygame CPU 粒子的性能上限（一般 \&lt; 2000 粒子）。

#### 重点理解

- 粒子四要素：生成器（位置/速率）、初始属性（速度/颜色/大小/寿命）、运动逻辑（重力/阻尼）、消亡回收。

- 批量管理：将所有粒子存在一个列表中，每帧更新并绘制，移除死亡粒子。

- 性能瓶颈：纯 Python 循环 \+ 绘制每个粒子会明显降低帧率。现代引擎使用 GPU 或 C\+\+ 后端。

- 为什么 GPU 更快：粒子属性可存储在 GPU 数组中，并行计算位置和生命周期。

#### 动手练习题

1. 实现 `Particle` 类（位置、速度、颜色、大小、寿命、衰减速度），每帧更新并淡出。

2. 实现 `ParticleEmitter`：控制发射频率（每帧累积时间，到达间隔则发射新粒子），随机速度和颜色。

3. 做出三种特效：火焰（向上发射，橙红）、爆炸（向外散射，快速消失）、雪花（飘落，轻微左右摆）。

4. 性能测试：发射 3000 粒子，记录帧率，思考优化方案（减少每粒子独立绘制、使用 subsurface 纹理）。

#### 🔁 映射引擎原理

- 打开 Godot，创建 `GPUParticles2D` 节点，观察其参数面板：初始速度、加速度、阻尼、颜色渐变、缩放渐变。

- 思考：你的手写粒子系统与 `GPUParticles2D` 在实现机制上有何本质不同？你能否模拟它的“颜色渐变”功能？

### 第 16 课：事件系统与模块解耦通信（发布\-订阅模式）

#### 学习目标

1. 解决模块紧耦合问题；

2. 手写事件总线（注册/发布/注销）；

3. 对应 Godot 信号机制底层原理。

#### 重点理解

- 紧耦合痛点：A 直接调用 B 方法，导致修改 B 时必须修改 A。

- 事件总线：发布者 `emit\(event\_name, data\)`，订阅者 `on\(event\_name, callback\)`，两者无直接依赖。

- 引擎对照：Godot 中的 `signal` 本质上是对象间的事件机制，但更绑定于特定对象；事件总线是全局的。

#### 动手练习题

3. 扩展支持事件优先级和一次性订阅。

#### 🔁 映射引擎原理

- 在 Godot 中使用 `signal` 进行解耦：例如一个自定义信号 `coin\_collected`。

- 对比：Godot 信号与你的全局事件总线在灵活性和类型安全方面有何不同？

### 第 18 课：性能优化基础（含空间划分、纹理图集）

#### 学习目标

1. 定位 Pygame 常见性能瓶颈；

2. 掌握缓存、空间划分、纹理图集、脏矩形等优化手段；

3. 建立“先实现后优化”的工程原则。

#### 重点理解

- Pygame 瓶颈：大量独立绘制调用、频繁 IO、O\(N²\) 碰撞检测。

- 空间划分：将世界划分为网格，只检测同一或相邻网格内的物体。例：200 个物体，全检测 4 万次，网格划分后可能仅数千次。

- 纹理图集：将多个小图合并为一张大图，用 `Surface\.subsurface` 切出所需区域，减少纹理切换（虽然 Pygame 是 CPU 渲染，但仍可减少 Surface 对象数量）。

- 其他优化：使用 `convert\(\)` 或 `convert\_alpha\(\)` 加速 blit；避免每帧创建新 Surface；限制粒子数量。

#### 动手练习题

1. 资源管理器缓存对比：重复加载图片 vs 一次缓存的帧率差异。

2. 实现简单网格空间索引（例如 64x64 网格），用于碰撞优化，测量 200 个随机运动物体的帧率提升。

3. 纹理图集：将 10 张小图标合并成一张大图，并通过 `subsurface` 分别绘制，测量 blit 性能差异（可选）。

4. （高级）脏矩形更新：仅更新鼠标移动影响的小区域，其它区域使用上次绘制结果。

#### 🔁 映射引擎原理

- 了解 Godot 的 `TileMap` 底层使用图集进行批量绘制。

- 研究：Godot 的物理引擎如何自动进行空间划分（BVH 或 Octree）。阅读 `PhysicsServer` 相关文档。
